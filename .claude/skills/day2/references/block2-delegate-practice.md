## [2] Delegate 실습
═══════════════════════════════

Delegate(위임)는 3단계로 합니다.
신입에게 업무를 맡기는 것과 동일한 원리입니다.

  📋 1단계: 기준 정의
  ─────────────────────────────
  "좋은 결과"가 어떤 모습인지 먼저 정한다
  예: "경쟁사 비교표는 5개 항목 × 3개 회사, 출처 포함"

  🤝 2단계: 위임
  ─────────────────────────────
  Day 1에서 배운 CO-STAR 또는 CRIT로 요청한다
  구조화된 요청 + 역질문 패턴 활용

  ✅ 3단계: 검증 + 피드백
  ─────────────────────────────
  결과를 내 기준으로 평가한다
  "이걸 팀장에게 그대로 보낼 수 있는가?"
  부족하면 구체적으로 피드백 후 재요청


💡 왜 기준 정의가 먼저인가?

기대치 없이 맡기면:
  "보고서 써줘" → 방향이 엉뚱한 결과 → 처음부터 다시

기준을 먼저 정하면:
  "5개 항목으로 비교표를 만들어줘" → 수정만 하면 됨 → 시간 절약

### 실제 업무로 Delegate 실습

Block 1에서 🤝 Delegate로 분류한 업무 중 1개를 골라
실제로 3단계를 적용해봅시다.

**1단계: 기준 정의**
먼저 "좋은 결과"가 뭔지 적어보세요:
- 어떤 형식? (표/문서/리스트)
- 어떤 내용이 포함되어야 하나?
- 누가 읽을 건가?
- 분량은?

**2단계: 위임 (Claude에게 요청)**
Day 1에서 배운 방법 중 하나를 사용합니다:

방법 A — CO-STAR 구조:
```
[C] 배경: ...
[O] 목표: ...
[S] 스타일: ...
[T] 톤: ...
[A] 대상: ...
[R] 응답 형식: ...
```

방법 B — CRIT 역질문:
```
[업무 설명]. 시작하기 전에 더 좋은 결과를 위해
핵심 질문 5개를 먼저 던져줘.
```

**3단계: 검증 + 피드백**
결과를 확인하고:
- "이걸 그대로 쓸 수 있는가?"
- 부족한 점이 있으면 구체적으로 피드백
- 피드백 후 재요청하여 개선된 결과 확인

💡 Before/After를 비교해보세요.
   첫 번째 결과와 피드백 후 결과가 얼마나 다른지 체감하는 게 핵심입니다.
